Именование объектов. Golang

Конспект статьи от Dave Cheney посвященный рекомендациям по именованию объектов.

Слабый нейминг есть симптом слабого дизайна - Dave Cheney

или еще вот:

Хороший нейминг как хорошая шутка. Если приходится объяснять, то уже не смешно — Dave Cheney

Хорошо если нейминг:

  • лаконичный, но краткость не вредит ясности;
  • описывает результат функции, а не ее реализацию;
  • позволяет без труда предугадать как и где объект используется.

Краткость vs ясность #

Чем больше разнесены в коде объявление и использование объекта, тем более длинным должно быть его имя - Andrew Gerrand

Подходы, которых следует придерживаться:

  • Короткие имена назначаюся если расстояние между объявлением и крайним использованием объекта минимально, например в цикле;
  • Использование длинного нейминга должно быть целесообразно;
  • Не включать в имя тип объекта - врядли он сгладит плохой нейминг;
  • Имя константы описывает значение, но не как его использовать;
  • Односимвольные имена внутри циклов;
  • В одно слово для параметров и возвращаемых значений функций, методов, интерфейсов и пакетов;
  • Многословные для названий функций и объявлений на уровне пакетов;
  • Название пакета так же является частью нейминга вызываемого объекта, используй это;
  • Не смешивать короткие и длинные имена в одном объявлении.

Пример:

type Person struct {
	Name string
	Age  int
}

// AverageAge returns the average age of people.
func AverageAge(people []Person) int {
	if len(people) == 0 {
		return 0
	}

	var count, sum int
	for _, p := range people {
		sum += p.Age
		count += 1
	}

	return sum / count
}
  • p - объявляется и используется в соседний строчках;
  • people, count, sum - объявляется и затем многократно используется;
    • если заменить count на c и sum на s, то это автоматически вынудит переименовать все переменные в программе. Врядли это добавит ясности.

Используй разделение кода пустыми строками так же как разделяется текст на абзацы. Каждый из них являет собой единое смысл, действие

Использование единообразных имен #

Имя объекта должно быть предсказуемым. Встречая его в первый раз, читатель должен понять его смысл.

Если далее по коду он встретит аналогичное имя, должна быть уверенность, что смысл объекта остался прежним.

Вместо d *sql.DB, dbase *sql.DB, DB *sql.DB обозначающих одно и тоже стоит выбрать единый, например db *sql.DB.

Тоже самое валидно и для получателей значений из функций и методов.

Традиционно односимвольные имена используются в циклах, где:

  • i (j) - индекс в цикле, j используется для вложенных циклов;
  • k, v - ключ/значение в map;
  • n часто ассоциируется со счетчиком (возврастает);
  • s для обозначения string.

Таким образом читатель встретив i или j будет предполагать, что он находится в цикле.

Если ты очнулся обложенным кучей i и j, то самое время задуматься над вынесеннием функционала в отдельные модули

Варианты объявления переменных #

  • Когда переменная дикларируется без инициализации (будет сделано позже), используй var:

    var players int    // 0

var things []Thing // an empty slice of Things

var thing Thing    // empty Thing struct
json.Unmarshall(reader, &thing)

  • Когда требуется в том числе и явная инициализация - используй :=
