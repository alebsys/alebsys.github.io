Resolve DNS записей в k8s

Дано #

  1. Развернутый coreDNS в кластере;
  2. Внешний по отношению к k8s DNS сервер, далее WinDNS;
  3. Произвольный Под в кластере из которого пойдет DNS-запрос.

Про CoreDNS #

CoreDNS представляет собой написанный на Go DNS-сервер, использующийся для Service Discovery в кластере.

Сервис модульный, в связи с чем весь функционал вынесен в отдельные плагины. В данном тексте сосредоточимся на kubernetes и forward плагинах:

.:53 {
    ...
    kubernetes cluster.local in-addr.arpa ip6.arpa {
       pods insecure
       fallthrough in-addr.arpa ip6.arpa
       ttl 30
    }
    ...
    forward . /etc/resolv.conf {
       max_concurrent 1000
    }
    ...
}

CoreDNS отвечает за кубовую зону cluster.local. Для резолва остальных доменов задействован плагин forward, конфигурация подтягивается из /etc/resolv.conf.

Про dnsPolicy #

Kubernetes позволяет на уровне Пода задавать dnsPolicy, определяя настройку DNS.

По умолчанию dnsPolicy выставляется в значение ClusterFirst - записывает в /etc/resolv.conf данные о kube-dns сервисе:

search namespace.svc.cluster.local svc.cluster.local cluster.local
nameserver 10.96.0.10
options ndots:5

Для Подов с CoreDNS на борту требуется другая настройка, ибо он должен знать куда обращаться при разрешении доменов сторонних (не cluster.local) зон.

Его настройка — dnsPolicy: Default, то есть в Под подтягивается /etc/resolv.conf хостовой машины:

nameserver 10.96.124.1
nameserver 10.96.124.2

Про ndots и FQDN #

FQDN (Fully Qualified Domain Name) считается таковым если на конце присутствует . . К примеру example.com не является FQDN, когда как example.com. уже полноправный FQDN.

Параметр ndots определяет сколько в доменном имени должно присутствовать . чтобы оно резолвилось как FQDN.

Т.о. если на конце доменного имени нет . и/или всего . меньше чем значение ndots, то в начале будут последовательно резолвиться доменное имя + суффикс из секции search и только в конце домен как есть.

Таким образом обеспечивается service discovery в кластере k8s, хотя и ценной более частых “промахов”.

Пример #

Запустим в одном из Подов команду dig example.com, будут сгенерированы следующие пакеты:

  1. попытка резолва example.com.namespace.svc.cluster.local
// 192.168.210.138 адрес Пода, 192.168.48.45 адрес Пода с CoreDNS
192.168.210.138 192.168.48.45 DNS 134 Standard query 0x28a9 A example.com.namespace.svc.cluster.local
192.168.48.45 192.168.210.138 DNS 227 Standard query response 0x28a9 No such name A example.com.namespace.svc.cluster.local SOA ns.dns.cluster.local
  1. попытка резолва example.com.svc.cluster.local
192.168.210.138 192.168.48.45 DNS 112 Standard query 0xe1f7 A example.com.svc.cluster.local
192.168.48.45 192.168.210.138 DNS 205 Standard query response 0xe1f7 No such name A example.com.svc.cluster.local SOA ns.dns.cluster.local
  1. попытка резолва example.com.cluster.local
192.168.210.138 192.168.80.33 DNS 108 Standard query 0x6f64 A example.com.cluster.local
192.168.80.33 192.168.210.138 DNS 201 Standard query response 0x6f64 No such name A example.com.cluster.local SOA ns.dns.cluster.local
  1. попытка резолва example.com
192.168.210.138 192.168.48.45 DNS 94 Standard query 0x9822 A example.com
// CoreDNS в соотвествии с плагином forward перенаправляет запрос в WinDNS 
// 10.96.70.12 - машиша с CoreDNS Подом на борту 
// 10.96.124.1 - машина с WinDNS
10.96.70.10 10.96.124.1 DNS 85 Standard query 0x3ffb A example.com OPT
10.96.124.1 10.96.70.10 DNS 101 Standard query response 0x3ffb A example.com A 93.184.216.34 OPT
// Отправка ответа целевому Поду от CoreDNS
192.168.48.45 192.168.210.138 DNS 124 Standard query response 0x9822 A example.com A 93.184.216.34
Таким образом для финального резолва нам потребовалось только на стороне DNS-клиента отправить 4 запроса, когда как всего было сгенерировано 10.

Кастомизация resolv.conf #

Директива dnsConfig на уровне Пода позволяет переопределять файл resolv.conf, меняя такие параметры как адреса nameserver’ов, величину ndots, суффиксы search и подобное. Больше подробностей в документации.
